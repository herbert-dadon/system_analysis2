איך הכי נכון להתכונן

המבחן הזה הוא כמעט כולו יישומי: נותנים לך תרחיש מילולי (כמו "חדר הכושר") ומבקשים לבנות ממנו סדרת תרשימים — מחלקות, Use-Case, פעילות, מצבים, רצף — ולענות על כמה שאלות מושגיות. לכן הכלל הראשון: ידע פסיבי לא מספיק, חייבים לתרגל בנייה ביד.

  • חומר פתוch ≠ פנאי. במבחן פתוח אין זמן לחפש מושגים — הדף נועד לאימות נוסחאות וסימונים, לא ללמידה. תכין לעצמך "דף סימונים" מרוכז (קשרים, ריבוי, צורות חצים) שתוכל לסרוק בשנייה.
  • שלוט בזרימת השאלה. שאלות המבחן בנויות בשרשרת: סעיף 1 תרשים מחלקות (נתונים בלבד) → סעיף תרשים רצף → חזרה לעדכן את תרשים המחלקות עם מחלקות ממשק ובקרה. הכר את הרצף הזה מראש.
  • היצמד לדרישות הטקסט. כל פונקציה/קשר/מחלקה צריך עוגן במשפט בשאלה. מותר להוסיף פונקציות הגיוניות, אבל קודם כל לכסות את מה שנכתב במפורש.
  • סמן מה שאתה שולף מהטקסט. תוך כדי קריאה — סמן ישויות (שמות עצם), קשרים (פעלים/"שייך ל"), אילוצי ריבוי ("מספר", "אחד או יותר"), ופונקציות ("יש לאפשר הוספה/עדכון/מחיקה").
  • תרגל את אותה דוגמה בכל התרשימים. קח תרחיש אחד (חדר כושר / חנות) ובנה עליו את כל סוגי התרשימים. ככה אתה לומד את הקשרים בין התרשימים, וזה בדיוק מה שהמבחן בודק.
לכל נושא כאן יש תיבת ✅ — סמן נושא כשאתה מרגיש שאתה יודע לבנות אותו לבד מתרחיש, לא רק להסביר אותו. בראש העמוד תראה את ההתקדמות.
חלק א

יסודות מונחה-עצמים

אבני הבניין: עצם, מחלקה, תכונה, מפתח, פונקציה — וחמשת סוגי הקשרים. זה הבסיס שכל שאר התרשימים נשענים עליו.

עצם (Object)

דבר הקיים במציאות — מוחשי או מופשט. ארבעה סוגים: עצם פיזי (בניין, ספר), גוף חי (סטודנט, עובד), מושג מופשט/רעיון (קורס, תעסוקה), אירוע (דיווח נוכחות, תנועה בחשבון).

תכונה (Attribute) — 3 סוגים

מחלקה (Class)

אוסף עצמים מאותו סוג — בעלי אותן תכונות, אותן פונקציות ואותם סוגי קשרים.

מפתח (Key)

תכונה אחת או יותר המזהה עצם במחלקה באופן חד-חד-ערכי (למשל ת"ז לתלמידים).

פונקציות מרכזיות (זכור כ-CRUD)

  • הוספה — יצירת עצם חדש (Add New Employee+)
  • חיפוש/שליפה — מקבלת מזהה ומאתרת. הבחנה: חיפוש מאשר קיום, שליפה גם מחזירה
  • עדכון/שינוי — לפי מה שנדרש בשאלה
  • ביטול/מחיקה — הסרה מהמערכת
מורכבת = רכיב אחד שמתפרק לתת-שדות שונים (כתובת → רחוב+עיר). סט = ריבוי ערכים מאותו סוג (כמה מספרי טלפון). השאלה היא: ערכים שונים זה מזה (מורכבת) או הרבה מאותו דבר (סט).
רק הפונקציות הרלוונטיות לדרישות השאלה. אם המערכת צריכה "להוסיף עובד" — מוסיפים +addNewEmployee(). לא ממציאים פונקציות שלא נדרשו, אבל כן מוסיפים את ה-CRUD ההגיוני שהתרחיש מרמז עליו.

קשר = יחס בעל משמעות (תלות סמנטית) בין מחלקות, מסומן בקו.

שלושת קשרי הריבוי הבסיסיים

  • 1:1 (נדיר) — עובד ↔ כרטיס אשראי אחד
  • 1:* (יחיד לרבים) — מחלקה אחת ↔ הרבה סטודנטים
  • *:* (רבים לרבים) — עובד ↔ פרויקטים
פירוק רבים-לרבים: כשיש קשר *:* וצריך לשמור תכונה משותפת (כמו "ציון" המשותף לסטודנט וקורס) — חייבים מחלקה אמצעית. סטודנט 1..* → ציון ← *..1 קורס. גם קשר *:*:* (שלושה) מתפרק למחלקה אמצעית (לקוח-מכירה-סוכן-מוצר).
שני סימנים: (1) הניסוח "כל X ל-מספר Y, וכל Y ל-מספר X"; (2) קיים נתון ששייך לקשר עצמו, לא לאחת המחלקות — כמו ציון (שייך לזוג סטודנט-קורס) או "שעות" (שייך לזוג עובד-פרויקט). הנתון הזה חייב לגור במחלקת הקשר.

חלק ג

מחלקות מתקדמות

מחלקה מופשטת ופולימורפיזם, ושלושת סוגי המחלקות במידול מערכת: ממשק, בקרה, נתונים (MVC).

מחלקה מופשטת = מחלקה שאי-אפשר ליצור ממנה עצמים, אבל יש לה תכונות, פונקציות ומחלקות-משנה שיורשות ממנה. מטרתה: חיסכון ומניעת כפילויות.

  • פונקציה מופשטת = רק הכרזה, ללא מימוש. המימוש קורה בתת-המחלקות (הגדרה מחדש).
  • שיטה מופשטת יוצרת "חוזה": כל יורשת חייבת לממש אותה — או להישאר בעצמה מופשטת.
  • אין חובת מפתח למחלקה מופשטת (אין לה עצמים להבחין ביניהם).
  • פולימורפיזם: אותה קריאה getArea() מתנהגת אחרת בכל יורשת (Rectangle/Triangle/Circle).
// מחלקת בסיס מופשטת
public abstract class Shape {
    protected Color color;
    public void setColor(Color c){ this.color = c; }
    public abstract double getArea();   // ללא מימוש
}
public class Rectangle extends Shape {
    protected double width, height;
    public double getArea(){ return width*height; } // מימוש מחדש
}
ניסיון ליצור עצם ממחלקה מופשטת = שגיאת הידור: class Shape is abstract; cannot be instantiated. אם נשאלת "האם זו מחלקה מופשטת?" — חפש מחלקת-על שמכילה רק הגדרות משותפות בלי עצמים משלה.
כי לעולם לא ניצור "צורה" סתמית — תמיד מלבן/עיגול/משולש קונקרטי. אבל לכולן משותפות תכונות (צבע, שטח, היקף) ופעולות (חשב שטח, צייר). מגדירים את המשותף פעם אחת ב-Shape ומחייבים כל יורשת לממש את getArea() בדרכה.

בניתוח מ"מ 1 הכרת רק מחלקות נתונים. בקורס הזה מוסיפים שתיים:

ממשק (Boundary)מה שהמשתמש רואה — טפסים, דו"חותטופס הזנת מוצר
בקרה (Control)תהליך/פעילות — מתווך בין ממשק לנתונים (שאילתות, לוגיקה)בקרת הזמנות

זרימת MVC: משתמש ← ממשק ← בקרה ← נתונים

כללי הקשרים שחייבים לזכור:
• ממשק ↔ בקרה = 1:1
• בקרה ↔ נתונים = 1:* (יחיד-לרבים), כשהבקרה בצד ה-"1".
• מחלקות הממשק והבקרה מכילות את הפונקציות שהמשתמש צריך, וחייבות להכיר את הפונקציות שימומשו במחלקת הנתונים.
מתי יוצרים מחלקות ממשק ובקרה? לא בסעיף תרשים המחלקות הראשון (שם רק נתונים). יוצרים אותן כשמגיעים לתרשים הרצף — ואז חוזרים ומעדכנים את תרשים המחלקות. שמרו על שמות זהים בין הממשק לבקרה לאותה ישות.
המשתמש מזין נתונים לטופס (ממשק) ולוחץ "שמור". הממשק מכיר את הפונקציה save ומעביר אותה לבקרה. הבקרה (המנהלת של מחלקת הנתונים) מעבירה את הפקודה למחלקת הנתונים שמבצעת בפועל ומחזירה OK חזרה במעלה השרשרת.
חלק ד

תרשימי UML

13 סוגי תרשימים ב-UML, מחולקים ל-3 קבוצות. אלה שבמבחן: מחלקות, עצמים, Use-Case, פעילות, מצבים, רצף.

קבוצהמידולתרשימים עיקריים
מבנייםסטטימחלקות, עצמים, רכיבים, פריסה, חבילות
התנהגותייםדינמיפעילות, מצבים, Use-Case
אינטראקציהדינמירצף, שיתוף-פעולה, תזמון
UML מאגדת טכניקות ותרשימים לייצוג ואפיון התנהגות מערכת המידע. כולל היום 13 סוגי דיאגרמות.

UC = רצף פעולות שהמערכת מבצעת כדי להפיק תוצאה עבור שחקן. מתאר פונקציונליות, לא תהליכים. הוא "סיפור" — התרשים מלווה בטקסט.

  • מלבן = גבולות המערכת. בתוכו האליפסות (UC), מחוץ לו השחקנים.
  • שחקן = גורם חיצוני עם אינטראקציה למערכת (אדם / מערכת אחרת / חומרה).
  • אליפסה = תרחיש שימוש בודד, עם שמו בפנים.
  • מחברים שחקן ל-UC רק כשיש אינטראקציה (קלט/פלט) איתו. פעולה אוטומטית = בלי שחקן.

שני קשרי התלות בין UC-ים

קשרמתימילות מפתח
<<include>> (שימוש)קשר הכרחי — ה-UC הבסיסי תמיד משתמש בו"בכל פעם", "תמיד"
<<extend>> (הרחבה)קשר אפשרי — מתבצע רק בתנאי"במידה ו...", "אם..."
המבחן לבחירה: שאל את עצמך — "האם ה-UC הבסיסי שלם בלי זה?" אם כן → extend (אופציונלי). אם הבסיסי לא יכול לעבוד בלעדיו → include (חובה). דוגמה: "וידוא כרטיס אשראי" ברכישה = include; "שדרוג רמת גימור" ברכישת רכב = extend.

פונקציונליות. כספומט = "משיכת כסף", "בירור יתרה" — ולא את צעדי התהליך הפנימי. את הצעדים מתארים בטבלת ה-UC או בתרשים רצף/פעילות.

רצף הצעדים/פעילויות המתאר תהליך במערכת — בעצם תרשים זרימה. מתאים ללוגיקות מורכבות (סיכום חשבון במסעדה, השאלת ספר, כניסה לבניין).

רכיבים

  • עיגול מלא = התחלה; עיגול בתוך טבעת = סיום.
  • מלבן מעוגל = פעילות/צעד.
  • מעוין = תנאי (החלטה) עם הסתעפות כן/לא.
  • קו עבה (fork/join) = פיצול לפעילויות מקבילות ואיחודן.
כשפעילויות מתבצעות במקביל ואינן תלויות זו בזו — מפצלים בקו עבה לכמה זרימות, ומאחדים בקו עבה נוסף לפני ההמשך.

מתאר התנהגות דינמית של אובייקט יחיד לאורך חייו — המצבים שהוא עובר והמעברים (Transitions) ביניהם בעקבות תרחיש/תנאי. נפוץ במערכות זמן-אמת (real-time).

תהליך בנייה

  • למד את התנהגות המחלקה
  • מדל לוגיקות התנהגות רגילות
  • הוסף לוגיקות התנהגות לא-רגילות (שגיאות, אזעקה)
  • עדן את המודל ובדוק מול תרשימי אינטראקציה ומחלקות
דוגמאות-עוגן מהמבחנים: כספומט (Idle→In Work→Waiting) ושער/קרוסלה ברכבת (נעול→פתוח). את הקרוסלה מרחיבים בהדרגה: התנהגות רגילה → התנהגות לא-רגילה (אזעקה בניסיון מעבר בנעול) → עידון (מצב "הפרת חוק" שנשבר רק ב-Reset).
מצבים = איך עצם בודד משנה מצב לאורך חייו (state→transition→state). פעילות = רצף צעדי תהליך שלם, אולי על פני כמה גורמים, עם תנאים והסתעפויות. מצבים מתמקד בישות; פעילות בזרימת התהליך.

תרשים מחלקות

הליבה של המבחן — מתאר את מחלקות התוכנה והקשרים ביניהן. כל מחלקה = מלבן ב-3 חלקים: שם / תכונות / פונקציות. + = public, - = private/protected.

תרשים עצמים (פחות נפוץ)

  • מופע של תרשים מחלקות — תמונה רגעית של נתוני המערכת.
  • כותרת העצם = שם_עצם : שם_מחלקה, ולכל תכונה ערך קונקרטי.
  • כלי לבדיקת תאימות המודל למציאות. לתרשים מחלקות אחד יש מספר תרשימי עצמים אפשריים.

מתאר את פעילות הגומלין בין העצמים לאורך ציר זמן, דרך מסרים (messages). חוליה מקשרת: ה-UC מציג פעילות, תרשים המחלקות מציג מחלקות עם פונקציות, ותרשים הרצף מראה את יחסי הגומלין ביניהן.

סימונים

  • מסר = חץ עם שם הפונקציה.
  • שחקן (Actor), מחלקת ממשק, מחלקת בקרה, אובייקט ספציפי (St : Student) או כללי (: Courses).
  • לולאה = מעבר מחזורי על מספר אובייקטים.
  • Ref = הפניה לתת-תרשים רצף (תהליך נפרד).

כללי ברזל בתרשים רצף

  • אפשר לשלוח מסר רק אם (א) יש קשר בין המחלקות בתרשים המחלקות, ו-(ב) המתודה היא public.
  • מחלקות ממשק מדברות עם מחלקות הנתונים אך ורק דרך מחלקות הבקרה.
  • כל פונקציה חייבת להיות מוכרת בשני הצדדים (אצל המחלקה שמקבלת).
  • התרשים מתחיל בשחקן שמאתחל אותו; פונקציה בודדת לאובייקט בודד (אלא אם לולאה).
  • יצירת אובייקט חדש לראשונה — מצוירת נמוך יותר, לפי התקדמות הזמן.
זה הסעיף שמייצר את מחלקות הממשק והבקרה! אם מבקשים תרשים רצף ל"הוספת לקוח חדש" — תיצור ממשק לקוחות ובקרת לקוחות בנוסף למחלקת הנתונים "לקוחות" שכבר קיימת, ואז תחזור לעדכן את תרשים המחלקות.
שחקן (Employee) → ממשק (Orders&Products) → בקרה (Orders&Products) → מחלקות נתונים (Product, Order, OrderItem). הממשק מעולם לא קופץ ישר לנתונים — תמיד דרך הבקרה. תהליך התשלום מופנה ל-Ref. Payment נפרד.

תבניות עיצוב לבעיות חוזרות בתכנות מונחה-עצמים. מטרות: שימוש חוזר, בהירות, וחיסכון בטעויות.

תבניתהבעיה שהיא פותרת
Prototypeאיך לייצר העתק של עצם פולימורפי שטיפוסו ידוע רק בזמן ריצה. פתרון: פונקציה וירטואלית Clone() שמחזירה העתק. יתרון: מודולריות + יעילות. חיסרון: לא ניתן להעביר פרמטרים.
Compositeמבנה רקורסיבי — מחלקה שמכילה מחלקות מאותו סוג. מחלקת composite מנהלת רשימת מחלקות פשוטות ופונה אליהן. שימושי בייצוג טקסטים וב-GUI.
Adapterגישור בין ממשקים לא-תואמים — מתאם שמאפשר לעצם מטיפוס B לעבוד עם עצמים שמצפים לטיפוס A.

מערכת דואר: מחלקת "שעון" צריכה "להודיע" על חצות למחלקות הודעות והודעות-לארכיון. דרישה: מודולריות ואי-תלות — לא רוצים שהשעון (ה-Subject) והמיודעים (Observers) יהיו קשורים בקוד. השעון מודיע, וכל מי שנרשם כ-Observer מגיב.

תרשים מחלקות

חדר הכושר — תרשים מחלקות מלא

חדר כושר שומר נתונים על עובדים, לקוחות, מנויים, סוגי מנויים ורכישות בחנות. חלק מהעובדים הם מדריכים (התמחות אחת או יותר). ללקוח יכולים להיות כמה מנויים. ניתן לבצע הזמנות למוצרים (ללא הגבלת כמות), ולשלם גם באשראי. נדרש לאפשר: הוספת לקוח, יצירת הזמנה, יצירת מנוי, בחירת/עדכון סוג מנוי, שינוי סטטוס הזמנה, הצגת מחיר כולל, בחירת מוצר, הוספת עובד.

ישויות (מחלקות נתונים): Employees, Customers, Members, MembersTypes, Instructors, Orders, OrderItem, Products, רשימת התמחויות, Shift schedule, Employees/Customers Attendance.

קשרים מרכזיים:
• Customer ↔ Member = 1:* (ללקוח כמה מנויים).
• MembersType ↔ Members = 1:* (סוג מנוי משותף להרבה מנויים).
• Order ↔ OrderItem = 1:* ; OrderItem ↔ Product = *:1.
• Instructor ↔ התמחות = *:* (מדריך כמה התמחויות, התמחות לכמה מדריכים) → דורש מחלקה אמצעית.
• Employee ↔ Instructor = הורשה (מדריך הוא סוג של עובד).

פונקציות לדוגמה: CreateNewCustomer(), CreateNewOrder(), ChangeOrderStatus(), CalculateTotalCharge(), UpdatePrice(), AddNewEmployee().

אל תשכח: בסעיף הזה — מחלקות נתונים בלבד. את הממשק והבקרה תוסיף כשתגיע לסעיף תרשים הרצף.
תרשים פעילות

חשבון במסעדה

מחשבים את סכום כל המנות. אם הלקוח חבר מועדון — 10% הנחה; ואם גם יום הולדת — מפחיתים את המנה שאכל. תמיד מוסיפים דמי אבטחה. אם הלקוח ביקש להוריד דמי אבטחה — מורידים שקל מכל סועד. בסיום מדפיסים את החשבון.

התחלה → "חשב סכום מנות" → תנאי חבר מועדון? → אם כן: הנחה 10% → תנאי יום הולדת? → אם כן: הפחת מנת הלקוח → "הוסף דמי אבטחה" → תנאי ביקש להוריד אבטחה? → אם כן: "הורד שקל × מספר סועדים" → "הדפס חשבון" → סיום.

מלכודת: כל "אם" בטקסט = מעוין תנאי עם הסתעפות כן/לא. "תמיד" = פעולה ללא תנאי.
תרשים מצבים

שער (קרוסלה) ברכבת — כולל התנהגות חריגה

השער נפתח בעת העברת כרטיס/מטבע והלקוח עובר. הרחבה: ניסיון מעבר במצב נעול → מפעיל אזעקה, והמערכת נכנסת ל"הפרת חוק". במצב זה היא נשארת גם אם מכניסים מטבע/מנסים לעבור — עד לחיצת Reset, שמחזירה למצב נעול.

מצבים: Locked (התחלתי) → [insert coin] → Unlocked → [pass] → Locked.
התנהגות חריגה: ב-Locked, [try to pass] → אזעקה → Violation. ב-Violation: [coin / try pass] → נשאר ב-Violation; [Reset] → Locked.

שיטה: בנה קודם את הזרימה הרגילה (locked↔unlocked), ורק אז הוסף את המצב החריג והמעברים אליו וממנו. זה בדיוק תהליך הבנייה ההדרגתי שהמרצה מלמד.
Use Case

רכישת מוצרים עם תשלום באשראי

מערכת המאפשרת ללקוחות לרכוש מוצרים, וניתן לשלם גם באשראי. בנה תרשים Use Case (התמקד בפונקציונליות).

שחקנים: לקוח, חברת אשראי (חיצונית).
UC עיקריים: "בחר מוצר", "בצע רכישה", "שלם".
• "בצע רכישה" <<include>> "שלם" (חובה — אין רכישה בלי תשלום).
• "שלם" <<extend>> "תשלום באשראי" (אפשרי — "במידה ו..." בחר אשראי).
• חבר את הלקוח ל-UC שיש בהם אינטראקציה. "וידוא כרטיס אשראי" מתחבר לחברת האשראי ולא ללקוח. פעולה אוטומטית (הדפסת קבלה) — ללא חיבור שחקן.
תרשים רצף

רכישת מוצר — מי מדבר עם מי

תאר בתרשים רצף את תרחיש רכישת המוצר בחדר הכושר. גזור מהשאלה אילו מחלקות ממשק ובקרה צריך להוסיף.

ציר השחקן Employee → Interface: OrdersAndProducts → Controller: OrdersAndProducts → מחלקות נתונים Product / Order / OrderItem.
רצף מסרים: ShowProductDetails() → (בקרה) GetProductDetails() (מ-Product) → CreateNewOrder() → ChooseProduct() → AddNewItem() → UpdateQty/Price() → ShowTotalCharge() → CalculateTotalCharge() → Pay() מופנה ל-Ref. Payment.

זכור: כל פונקציה ששלחת חייבת להופיע במחלקה המקבלת בתרשים המחלקות, והממשק לעולם לא ניגש ישירות לנתונים.
הורשה / מחלקה מופשטת

צורות גיאומטריות

שמירת צורות (משולש, מלבן, עיגול) עם צבע, היקף, שטח. למשולש: קודקוד, בסיס, גובה; למלבן: אורך, רוחב; לעיגול: רדיוס, מרכז. המערכת מחשבת היקף/שטח, מדפיסה פרטים ומציירת.

מחלקת-על מופשטת Shape: תכונה משותפת color, פונקציות computeArea(), computePerimeter(), draw(), printDetails() — חלקן מופשטות.
יורשות: Rectangle(אורך,רוחב), Triangle(בסיס,גובה,קודקוד), Circle(רדיוס,מרכז) — כל אחת מממשת מחדש את computeArea() (פולימורפיזם).
חץ הורשה חלול מכל יורשת אל Shape.

טיפים להצלחה

  • הכן "דף סימונים" אישי. צורות החצים (הקבצה/הרכבה/הורשה), סימני ריבוי, ומבנה טבלת ה-UC — שיהיו מול העיניים. במבחן פתוח חוסכים שניות יקרות.
  • קרא את כל השאלה לפני שמתחילים לצייר. סעיף הרצף יכריח אותך להוסיף מחלקות ממשק ובקרה לתרשים המחלקות — תכנן מקום מראש.
  • סמן בטקסט תוך כדי קריאה. שמות עצם = מחלקות; "שייך/כל...מספר" = קשרים וריבוי; "יש לאפשר..." = פונקציות.
  • עקביות שמות. מחלקת הממשק והבקרה של אותה ישות יקבלו שם זהה (Interface: Orders / Controller: Orders).
  • כל פונקציה צריכה בית. פונקציה שנשלחת בתרשים רצף חייבת להופיע במחלקה המקבלת בתרשים המחלקות — ושתי המחלקות חייבות להיות מחוברות.
  • הקבצה מול הרכבה — שאל "זכות קיום". אם החלק שורד בלי השלם → הקבצה (חלול). אם לא → הרכבה (מלא).
  • include מול extend — שאל "חובה?". תמיד מתבצע → include. רק "במידה ו..." → extend.
  • בתרשים מצבים בנה בשכבות. קודם התנהגות רגילה, אחר כך חריגות (אזעקה/הפרת חוק), בסוף עידון. אל תנסה הכל בבת אחת.
  • היצמד לדרישות, אבל אל תפחד להוסיף הגיוני. מותר להוסיף פונקציה סבירה; עדיף "יותר מדי הגיוני" מאשר לפספס דרישה מפורשת.
  • תרגל מבחן 2013 שלם בתנאי אמת. זה המקור שהמרצה בנה עליו את הקורס — מבני השאלות חוזרים.
